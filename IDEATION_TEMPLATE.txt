## Project Description

### Detailed description of the project and how you are incorporating the **Viral Mobile-Only Consumer Apps** and **x402 Agent Economy** tracks:

#### 1. What problem are you solving?

**Consumer Problem:**
Digital privacy is dead. When you share a private message, meme, or photo‚Äîyou lose all control. Recipients can screenshot, forward, and leak your content without consequences. Current solutions (Snapchat's screenshot detection, disappearing messages) are trivial to bypass and provide no **cryptographic guarantees** or **verifiable enforcement**.

**For the Agent Economy:**
As autonomous agents begin transacting in digital marketplaces, they need **self-enforcing digital rights** for the content they buy/sell. Agents can't rely on human legal systems‚Äîthey need smart contracts that enforce permissions autonomously and provide cryptographic proof of compliance.

#### 2. What insight led you to this problem?

**Consumer Insight:**
We observed that Gen Z users want **ephemerality + control**:
- Self-destructing content (Snapchat, BeReal)
- But with *real* enforcement (not spoofable)
- Gamified social features (streaks, challenges)
- Web3 ownership (your data = your asset)

**The killer combo:** Make privacy violations **expensive and public**. When someone tries to screenshot your RDO, it locks on-chain and alerts your followers. This creates **social pressure** and **viral sharing** ("Did you see John got locked for leaking Sarah's RDO?").

**Agent Economy Insight:**
AI agents will be the biggest consumers of digital content (training data, API access, media assets). They need:
- Automated payment rails (x402 SDK)
- Self-enforcing licensing (RDOs)
- Anomaly detection (PoEC) to avoid scam content

#### 3. What's your solution (tech overview)?

**RDO Protocol** = **Mobile-First Consumer App** + **Backend Agent Infrastructure**

### üì± **Part 1: Viral Mobile Consumer App**

**Core Features:**
1. **"Snapchat for Web3"** - Send self-destructing messages/photos as RDOs
2. **One-Time View Objects** - Content that self-destructs after first read (can't be forwarded)
3. **QR Code Sharing** - Scan to unlock secret content (physical‚Üídigital viral loop)
4. **Violation Leaderboards** - Public shame for users who violate rules (gamification)
5. **Creator Monetization** - Sell limited-access content (100 views max, auto-expire after 24h)

**Why It Goes Viral:**
- **Network Effects**: More users = more valuable secrets to unlock
- **FOMO**: Limited-use RDOs create scarcity ("Only 5 people can see this")
- **Gamification**: Compete for "most trusted user" (fewest violations)
- **Meme Culture**: Viral RDO challenges ("Send me an RDO I can't leak")

### ü§ñ **Part 2: x402 Agent Economy Backend**

**Agent Use Cases:**
- **Autonomous Data Marketplaces**: Agents buy/sell datasets as RDOs
- **AI Model Licensing**: GPT-4 wrapper sold as RDO (pay-per-inference via x402)
- **API Access Tokens**: RDOs that grant temporary API keys to agents
- **PoEC Risk Scoring**: GNN model flags suspicious agent wallets before purchase

### üîí **Part 3: PoEC - AI Compliance Layer**

**Graph Neural Network** detects anomalous behavior:
- Consumer: Warns you if recipient has leaked RDOs before
- Agent: Flags wallets with suspicious transaction patterns
- Generates **Merkle proofs** for on-chain verification

**Tech Stack:**
- **Smart Contracts (Solidity)**: Store rule hashes, track violations
- **IPFS**: Encrypted content storage
- **AES-GCM**: Client-side encryption (keys never leave device)
- **GNN (PyTorch ‚Üí ONNX)**: Browser/mobile-based anomaly detection
- **x402 SDK**: Agent payment rails
- **Monad Blockchain**: High-performance EVM-compatible chain for fast consumer transactions

### Backend: Agent-Native Infrastructure

#### **Smart Contracts (Solidity on Monad)**

**Why Monad:**
- **10,000+ TPS**: Instant RDO mints (no gas wars, sub-second UX)
- **EVM-Compatible**: Drop-in x402 SDK support
- **Low Fees**: Consumer-friendly gas costs (< $0.01 per RDO)

**Contract: `RDORegistry.sol`**
```solidity
function createRDO(
    bytes32 rulesHash,
    uint8 rdoType,
    Rules memory rules,
    string metadataCID,
    address[] whitelist
) returns (uint256 rdoId)

function requestAction(
    uint256 rdoId,
    uint8 actionType,
    bytes signature
) // Triggers PoEC verification
```

**Events:**
- `RDOCreated(id, creator, type)`
- `ActionRequested(id, user, action)`
- `ViolationRecorded(id, violator, penalty)`

#### **x402 SDK Integration**

**Agent Payment Flows:**
1. Agent discovers RDO in marketplace
2. Calls `x402.pay(amount)` ‚Üí triggers `requestAction(READ)`
3. PoEC verifies agent wallet (GNN check)
4. If safe, contract emits `AccessGranted` event
5. Frontend decrypts content for agent

**Revenue Sharing:**
- Creator: 90% of payment
- PoEC operators: 5% (staking rewards)
- Protocol: 5% (treasury)

#### **PoEC: Graph Neural Network Anomaly Detection**

**How It Works:**
1. **Build Graph**: Fetch user's transaction history from Monad
2. **Extract Features**: 20+ metrics (degree centrality, clustering, gas patterns)
3. **GNN Inference**: 2-layer GCN (64‚Üí32 neurons) ‚Üí anomaly score
4. **Merkle Proof**: Hash results + blockchain state ‚Üí on-chain verification

**Consumer Use Case:**
- Before unlocking RDO, check if recipient has violations
- Show risk score: üü¢ Safe (0-20%), üü° Caution (20-60%), üî¥ Risky (60%+)

**Agent Use Case:**
- Flag wallets with suspicious patterns (wash trading, sybil attacks)
- Auto-reject high-risk agents (or charge premium)

**Tech:**
- **Python Training**: PyTorch GNN trained on Monad testnet data
- **ONNX Export**: Convert to browser-compatible WebAssembly
- **Client-Side Inference**: No backend API (privacy + speed)

### Storage & Encryption

**IPFS (Pinata):**
- Encrypted content uploaded to IPFS
- Metadata JSON includes: `rulesHash`, `encryptedContentCID`, `iv`

**AES-GCM Encryption:**
- 256-bit keys generated client-side
- Content encrypted in browser/mobile before upload
- Decryption key stored in RDO metadata (only accessible if rules pass)

**Viem Cryptography:**
- Keccak256 hashing for deterministic rule compilation
- Ensures same rules ‚Üí same hash (on-chain verification)

## How the Project Works Technically

### **Consumer Flow: Creating a Self-Destructing RDO**

```
1. User opens mobile app ‚Üí taps "Create RDO"
2. Selects type: MESSAGE, PHOTO, VIDEO, LINK
3. Captures content (camera or text input)
4. Sets rules:
   - Max views: 1 (self-destruct after first read)
   - Forbid screenshot: true
   - Expiry: 24 hours
5. App generates AES-256 key ‚Üí encrypts content locally
6. Uploads encrypted blob to IPFS ‚Üí gets CID
7. Compiles rules ‚Üí Keccak256 hash
8. Signs transaction to Monad RDORegistry.createRDO()
9. Contract mints RDO, emits event
10. App generates QR code with RDO ID
11. User shares QR at party ‚Üí recipient scans ‚Üí unlocks secret
```

### **Consumer Flow: Accessing an RDO (with PoEC Check)**

```
1. Recipient scans QR ‚Üí opens mobile app
2. App calls RDORegistry.requestAction(rdoId, READ)
3. **PoEC Verification Triggered:**
   a. Fetch recipient's transaction history from Monad
   b. Build graph of wallet interactions
   c. Run GNN inference ‚Üí anomaly score
   d. If score > 60%, show warning: "High-risk user!"
4. If recipient approves (or score is safe):
   a. Smart contract checks rules (not expired, views < max)
   b. Emits AccessGranted event
5. App fetches IPFS metadata ‚Üí decrypts AES key
6. Downloads encrypted content ‚Üí decrypts locally
7. Displays content with countdown timer
8. After 10 seconds ‚Üí auto-deletes (memory cleared)
9. If user screenshots ‚Üí device detects ‚Üí triggers ViolationRecorded()
```

### **Agent Flow: Buying Dataset RDO with x402**

```
1. AI agent browses RDO marketplace API
2. Finds "GPT-4 Prompts Dataset" RDO (1000 uses, $10 in MONAD)
3. Agent wallet calls x402.pay(10 MONAD, rdoId)
4. Payment triggers PoEC verification:
   - GNN checks agent wallet for wash trading
   - Validates agent has no prior violations
5. If safe, RDORegistry.requestAction(DOWNLOAD) succeeds
6. Agent receives decryption key via API
7. Downloads dataset from IPFS
8. Uses data for model training
9. Agent's usage logged on-chain (transparency)
```
